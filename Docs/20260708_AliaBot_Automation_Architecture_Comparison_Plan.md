# 📋 AliaBot Automation Architecture: Cloud VM vs. Local PC Runner
## 모바일 지시의 백그라운드 자동 실행(Automation)을 위한 아키텍처 비교 계획서

본 문서는 AliaBot을 통해 모바일에서 전달된 지시사항을 사용자가 사무실에 복귀하기 전, AI 에이전트가 백그라운드에서 스스로 분석 및 연산을 완료해 놓도록 만드는 **전자동화(Automation) 아키텍처** 설계안입니다. **1안: 클라우드 가상 머신(Cloud VM)** 방식과 **2안: 상시 기동 사무실 컴퓨터(Local PC Runner)** 방식의 절차와 복잡도를 상세히 비교합니다.

---

## 1. ⚙️ 핵심 개념 및 작동 원리 (Terminology & Mechanism)

### ① 셀프 호스팅 (Self-Hosting - 자가 장비 운영 방식)
* **개념**: 외부 업체의 가상 컴퓨터를 임대하는 대신, 개인이 소유한 물리 컴퓨터(사무실 데스크탑)를 24시간 켜두고 외부 통신망에 서버로 연결하여 연산 장치로 직접 운용하는 형태입니다.

### ② 클라우드 호스팅 (Cloud Hosting - 가상 서버 임대 방식)
* **개념**: 아마존 AWS(Amazon Web Services) 또는 구글 클라우드(Google Cloud Platform)의 데이터 센터 내에 위치한 가상 컴퓨터(Virtual Machine)의 자원을 필요한 만큼 임대하여 24시간 가동하는 방식입니다.

### ③ 푸시 및 폴링 통신 (Push & Polling Communication)
* **개념**: 지시가 왔을 때 즉각 신호를 받아 기상하는 **푸시(Push)** 방식(예: GitHub Webhook)과, 1~5분 간격으로 새 편지가 왔는지 우체통을 주기적으로 열어보는 **폴링(Polling)** 방식의 작동 원리입니다.

---

## 2. 📊 아키텍처 구현 방식 비교 일람표

| 비교 항목 | 1안: AWS/구글 클라우드 가상 머신 (Cloud VM) | 2안: 상시 기동 사무실 컴퓨터 (Local PC Runner) |
| :--- | :--- | :--- |
| **개요** | AWS EC2 / GCP Compute Engine 상시 기동 | 24시간 켜져 있는 데스크탑 1을 에이전트로 등록 |
| **작동 원리** | 깃허브 웹훅 신호를 클라우드가 받아 즉시 연산 | 깃허브 액션(Actions) 셀프 러너가 신호 받아 PC 기동 |
| **월 고정 비용** | 💸 약 1.5만 원 ~ 5만 원 (사양 및 트래픽 연동 기준) | 🆓 0원 (기존 사무실 PC의 상시 전력만 사용) |
| **구축 난이도** | **높음 (High)**: Linux 서버 보안, 클라우드 권한 설정 필요 | **보통 (Medium)**: GitHub에 PC 등록 스크립트 실행 |
| **데이터 연동** | **복잡**: 로컬 영농 데이터를 클라우드로 수시 업로드 필요 | **매우 간편**: 이미 로컬 하드에 저장된 원시 데이터 즉시 가공 |
| **보안성** | 방화벽(Firewall) 및 암호화 설정 등 추가 관리 필요 | 회사/농장 내부망 안에서 돌기 때문에 물리 보안 탁월 |
| **실행 즉시성** | 지시 입력 시 수초 내 즉각 연산 시작 | 지시 입력 시 1분 이내에 자동 연산 시작 |

## 3. 🚦 방식별 상세 구현 절차 (Deployment Procedures)

### 🚀 1안: 클라우드 가상 머신 (Cloud VM - AWS EC2) 연동 절차

```
[AliaBot 지시] ──> [GitHub Webhook] ──> [AWS EC2 Linux 인스턴스] ──> [S3 스토리지 데이터 로드] ──> [결과 푸시]
```

1. **인스턴스 생성**: AWS 콘솔에 접속하여 Ubuntu Linux 기반의 가상 머신(EC2 Instance)을 개설합니다.
2. **방화벽 및 보안 그룹 설정**: 외부 깃허브 웹훅 신호와 HTTPS 통신만 수신하도록 포트(Port 443/80) 보안 규칙을 통제합니다.
3. **데이터 동기화망 구축**: 사무실 PC의 영농 크롭 데이터를 클라우드 가상 머신도 읽을 수 있도록 AWS S3 스토리지 또는 Dropbox/OneDrive 공유 드라이브와 실시간 클라우드 공유 폴더를 셋업합니다.
4. **감시 스크립트(Daemon) 배포**: Node.js 또는 Python 웹 서버를 띄워 깃허브 웹훅 신호를 24시간 감시하다가, 신호가 오면 크롭 데이터 연산 코드를 자동 구동하고 깃허브에 결과를 커밋하게 만듭니다.

---

### 💻 2안: 사무실 컴퓨터 직접 연동 절차 (GitHub Actions Runner)

```
[AliaBot 지시] ──> [GitHub Actions 서버] ──> [사무실 PC 내 Runner 에이전트] ──> [로컬 하드 데이터 직접 가공] ──> [결과 푸시]
```

1. **상시 기동 셋팅**: 사무실 컴퓨터(데스크탑 1)가 자동으로 잠자기 모드로 들어가지 않도록 윈도우 전원 관리를 `고성능(절전 없음)`으로 설정합니다.
2. **셀프 호스티드 러너 등록**:
   * 대상 GitHub 저장소의 `Settings` ➡️ `Actions` ➡️ `Runners` 메뉴로 들어갑니다.
   * `New self-hosted runner` 버튼을 클릭하여 제공되는 파워쉘(PowerShell) 환경 전용 연결 코드를 복사합니다.
3. **컴퓨터에 에이전트 설치**:
   * 사무실 컴퓨터 터미널에서 위 코드를 붙여넣어 실행하면, 컴퓨터 내부에 **GitHub Actions Runner 에이전트** 프로그램이 설치 및 백그라운드 구동됩니다.
4. **자동화 워크플로우(Workflow) 파일 작성**:
   * 프로젝트 내에 `.github/workflows/aliabot-runner.yml` 파일을 만들고, "AliaBot 지시사항 파일이 커밋되면, 이 작업을 `self-hosted` PC(사무실 컴퓨터)에서 실행하라"는 규칙 한 줄을 적어 깃허브에 올립니다.
   * 이제 밖에서 모바일로 지시를 내리면, 깃허브가 인터넷을 경유해 켜져 있는 사무실 컴퓨터에 직접 연산 지침을 쏘고 결과물을 생성하여 푸시하게 됩니다.

## 4. 💡 결론 및 최종 추천 경로 (Recommendation)

### "사무실 컴퓨터가 상시 켜져 있다면, 무조건 2안(Local PC Runner)이 유리합니다."

1. **비개발자 친화성**:
   * AWS 가상 머신(EC2)을 개설하면 리눅스 명령어(Linux Command Line)나 네트워크 보안 그룹(Security Group) 관리 등 비개발자가 다루기 어려운 장벽이 무척 많습니다.
   * 반면, **GitHub Actions Runner(2안)**는 깃허브가 제공하는 파워쉘 스크립트 실행 한 번만으로 내 PC가 서버처럼 동작하므로 설치 장벽이 현저히 낮습니다.
2. **대용량 영농 데이터 이동의 한계 극복**:
   * 1안은 매월 데이터 전송량 및 저장 용량에 따라 클라우드 비용이 폭증할 우려가 있지만, 2안은 로컬 C드라이브 하드디스크의 자원을 그대로 무제한으로 사용하므로 가장 경제적이고 타당합니다.
